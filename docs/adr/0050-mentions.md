# 0050. @提及 Mention（NIP-01 p-tag，密文內攜帶）

- 狀態：已接受
- 日期：2026-07-06
- 相關文件：`docs/ROADMAP.md`（Phase E @提及）、ADR-0011（reaction e-tag 引用）、ADR-0027（群組成對扇出）、ADR-0049（組織群組/公告）、ADR-0037（composer 尾端建議列）

## 背景與問題

企業組織群與一般群組需要「點名某人」的能力：讓被提及者收到通知、在訊息串中被凸顯。Nostr 既有 `p` tag（NIP-01）即表示「指涉某公鑰」，reactions/deletions 已以同源的 e-tag 引用機制運作。問題在於：

1. 提及資訊放哪？放在**明文外層**會讓中繼看到「誰提及誰」的社交圖譜（隱私洩漏）。
2. 如何從 composer 的 `@名稱` 對應到公鑰、且對顯示名稱變動穩健？
3. 被提及的通知與凸顯如何在收發兩端一致呈現？

## 考量的選項

- **選項 A（採用）：`p` tag 攜帶於加密 rumor 內層。** 提及＝rumor 內加 `["p", pubkey]`；隨 NIP-17/59 Gift Wrap 加密，只有群成員解得。UI 以候選清單（群成員/聯絡人）解析 `@名稱`→公鑰後傳入；收端讀 rumor p-tag 判定「是否提及我」。
- 選項 B：提及放外層 wrap tag。被中繼看見社交關係，違反隱私鐵則，否決。
- 選項 C：純文字約定（`@名稱` 不帶結構）。無法穩健對應公鑰、跨語言/改名即失效、無法可靠通知，否決。

## 決策

1. **提及 = 加密 rumor 內層 `p` tag。** `wrapMessage`／`wrapGroupMessage` 新增 `mentions?: PubkeyHex[]` 選項，去重後併入 rumor tags（`["p", pk]`）。與現有 `g`/`expiration`/`relay` tag 同層，維持「結構在密文內、比公開群聊更私密」。
2. **解析在客戶端、以候選清單為準（純函式）。** core `mention.ts`：`parseMentions(text, candidates)` 以 `@名稱` 掃描（名稱最長優先、詞界結尾、拉丁大小寫不敏感）回傳公鑰；`mentionedPubkeys(rumor)`／`isMentioned(rumor, self)` 供收端判定。名稱→公鑰對應由 UI 提供（群成員名冊／聯絡人），對顯示名稱以「送出當下」快照，事後改名不影響已送提及。
3. **收端行為：** 群訊解出 rumor 後若 `isMentioned` 為真，於 `ChatMessage.mentionsMe` 標記 → UI 凸顯該訊息，並在啟用通知時發桌面通知（即使該群目前非開啟視窗）。
4. **composer 體驗（沿用 ADR-0037 尾端建議列）：** 輸入 `@` 起頭的 token 觸發成員建議列，Tab/點擊補全為 `@顯示名稱`；送出時以 `parseMentions` 還原公鑰。群組候選＝群成員；1:1 候選＝對方（提及意義小，仍一致支援）。

## 理由

- 複用既有 NIP-01 `p` tag 與 Gift Wrap 加密，零新事件類型、零中繼改動、隱私不退讓。
- 與 reactions/deletions 的引用機制同源，收端解析與扇出路徑不變（ADR-0011/0027）。
- 解析為純函式，易測、與 UI/後端解耦；名稱快照避免改名造成的歧義。

## 後果

- 正面：企業組織群「點名＋通知」到位；社交圖譜不外洩；與現有加密/扇出/持久化自然相容（提及訊息一樣可回應/收回/限時）。
- 負面 / 已知殘餘風險：`@名稱` 若與成員顯示名稱含空白/重名，採「最長名稱優先」啟發式，極端重名仍可能誤配——由 composer 明確選取降低風險；被提及者若不在候選清單（非成員）不會被標記。
- 後續行動：composer 成員建議列與凸顯樣式（UI 增量）；未來可加「僅顯示提及我的」過濾與 @所有人（announce 群語意）。
