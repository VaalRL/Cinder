# 0122. 瀏覽器版重載會把你換成另一個人——身分不得被靜默替換

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0119（我在那裡拿掉了明文 nsec，卻沒給它替代的落腳處）**、
  0112（web/mobile 靜態加密）、0067（本地密碼）、0073（忘記密碼救援）、
  0117（行動端「記住我」——本 ADR 的樣板）、0053（OS 金鑰庫）

## 背景與問題

清點待辦時追一條線索，結果撞到一個**活的 P0**。實測（不是推論）：

```
原本的 pubkey : a1842467335935a5…
重載後 pubkey : e70c317c78da8357…
身分是否相同   : false
落盤的 identity: {"nsec":"nsec1rpmusggk2mzj2…   ← 而且是明文
```

**在瀏覽器版按一下重新整理，你就變成另一個人。** 聯絡人與訊息全部讀不出來（它們用舊金鑰
導出的 DEK 加密），而且系統靜默產生一把新 nsec **明文寫進 localStorage**——
再一次違反 ADR-0112 的紅線。

### 這是我自己在 ADR-0119 造成的

在 ADR-0119 之前，瀏覽器的 nsec 是**明文躺在 localStorage** 裡的。不安全，但重載後撈得回來，
所以「能用」。我把明文拿掉了（那是對的），**卻沒給它一個替代的落腳處**。

### 根因：ADR-0112 的「瀏覽器本地密碼」從來沒接線

| 零件 | 狀態 |
|---|---|
| `browserPassEnable()` / `browserPassUnlock()`（Argon2id 包裹） | 寫好了 |
| `browserKeyVault`（拒收明文 nsec） | 寫好了 |
| ADR-0119 修好的「瀏覽器解鎖」分支 | 寫好了 |
| **設定裡的「啟用本地密碼」** | **`if (!p || !isTauri()) return {};`——瀏覽器看不到** |
| **開機時檢查 `active.locked`** | **`if (isTauri() && active.relayUrl)`——瀏覽器不檢查** |
| **`browserPassEnable()` 的呼叫端** | **0 個** |

零件全都在，**沒有一根線接上去**。所以瀏覽器版沒有任何方式可以記住身分，而重載後
唯一的出路（貼回 nsec）也不存在——桌面的 `SignIn` **根本沒有 nsec 貼入欄位**。

### 而且使用者從沒看過那把 nsec

瀏覽器首次登入時 nsec 是在本機**產生**的，不會顯示給使用者。所以就算我們補上「貼回 nsec」
的入口，他手上也沒有東西可貼。**這個死結必須在登入的那一刻就解開，不能留到重載之後。**

## 決策

### 1. 🔴 安全網：**已知期待的身分時，絕不產生新身分**

這是本 ADR 最重要的一條，其餘都是它的配套。

```ts
// RelayChatBackend
let identity = storage.loadIdentity();
if (pool?.nsecOverride) { … }
else if (!identity) {
  // 呼叫端已經知道「這應該是誰」，卻拿不到金鑰 → **大聲失敗**。
  // 靜默產生新身分＝把使用者換成另一個人，而且會覆蓋掉他原本的資料。
  if (pool?.expectPubkey) throw new Error(IDENTITY_UNAVAILABLE);
  …generateSecretKey()
}
…
if (pool?.expectPubkey && pubkey !== pool.expectPubkey) throw new Error(IDENTITY_MISMATCH);
```

App 一律傳 `expectPubkey: active.pubkey`。於是**任何原因**——沒有 nsec、錯的 DEK、
儲存毀損、未來某個沒想到的 bug——都會變成一個明確的錯誤，而不是靜默換人。

**「安靜地做錯的事」是這個專案反覆出現的失敗模式**（ADR-0119 整份都在講這個）。
這條守衛是對它的一般性防禦，不只修今天這個 bug。

### 2. 瀏覽器首次登入**必填本地密碼**

不是選填。因為：

- nsec 是本機產生的，**使用者沒看過**；
- 瀏覽器的重新整理是隨手會做的事，不是「換裝置」這種罕見事件；
- 沒有密碼 ＝ 沒有任何方式可以把身分帶過重載。

所以登入畫面在瀏覽器模式多一個**必填**的密碼欄（Tauri 不顯示——那裡有 OS 金鑰庫）。
密碼以 Argon2id 包裹 nsec 落 localStorage；**磁碟上只有密文，KEK 從不落盤**（ADR-0112/0117）。

「新增身分」（多身分，ADR-0045）在瀏覽器同樣必填——理由完全相同。

### 3. 接上設定裡的「本地密碼」與開機解鎖

- 設定的安全區塊拿掉 `isTauri()` 閘門，改為**分流**（原生走 `invoke`，瀏覽器走 JS 的 Argon2id）。
- 開機時瀏覽器也檢查是否有包裹過的 blob → 有就走**解鎖畫面**（ADR-0119 已修好那條分支，
  但它一直打不到）。

**「停用密碼」在瀏覽器＝忘記這個身分**（刪掉 blob），而不是像桌面那樣把明文寫回金鑰庫。
瀏覽器**沒有**一個安全的明文去處——那正是 ADR-0112 的整個前提。

### 4. 忘記密碼的出路：**貼回 nsec**（重用既有的救援 UI）

`UnlockScreen` 已有 `onRescue: (nsec, newPassword) => Promise<boolean>`。桌面的實作是
ADR-0073 的雙重包裹救援；瀏覽器的實作更簡單：**驗證 nsec 的 pubkey 相符 → 以新密碼重新包裹 →
進入**。同一個 UI、同一個契約，不另開畫面。

要用這條路，使用者手上得有 nsec——設定裡的「身分備份」本來就有（複製 nsec）。

### 5. `SignIn` 補上「用 nsec 登入」

給兩種人用：**（a）** 忘記密碼又清掉了 blob 的人；**（b）** 在 ADR-0119～0122 之間卡住的
既有瀏覽器使用者（他們的身分已經被換掉了，但舊資料還在原 namespace 裡加密著——
貼回原本的 nsec 就能解開）。

## 理由

- 這個 bug 的形狀值得記住：**我修好了一個安全問題（明文私鑰），卻沒有處理「拿掉之後，
  它該住在哪裡」。** 安全修正如果讓功能無法運作，使用者只會退回更不安全的做法
  ——或者像這次，直接失去資料。
- 而它之所以能活下來，是因為**沒有任何測試會發現**：所有 UI 測試都是 SSR
  （`useEffect` 從不執行），而開機路徑正是一個 `useEffect`。所以本 ADR 的測試
  **不測 UI，測那條路徑的實質**：以真的 `LocalStorage` ＋ 真的 `RelayChatBackend`
  跑「寫入 → 重載 → 讀回」。

## 後果

- 正面：
  - **瀏覽器版可以重新整理了**（這聽起來很低的門檻，但它現在真的不行）。
  - 身分**不可能**被靜默替換——`expectPubkey` 讓它變成一個明確的錯誤。
  - ADR-0112 的瀏覽器靜態加密與 ADR-0119 的解鎖分支，**終於真的被接上**（原本全是死碼）。
  - 桌面（Tauri）行為完全不變——它有 OS 金鑰庫，不需要這些。
  - 測試 981 → **994**（engine +4：拿不到金鑰時拋錯且**什麼都不寫**、pubkey 不符拋錯、
    正常路徑不受影響、未傳 expectPubkey 沿用舊行為；desktop +9：**登入→重載→回得來**的
    端到端、磁碟上只有密文、密碼錯誤、停用＝忘記身分但資料還在、別人的 nsec 進不來、
    SignIn 的密碼欄與 nsec 出路）。

- 已知限制：
  - **既有的瀏覽器使用者需要貼回 nsec**（若他們沒備份，資料無法救回——DEK 由 nsec 導出，
    沒有第二道包裹）。這是 ADR-0119 造成的傷害，只能提供出路，無法自動修復。
  - 瀏覽器的「停用密碼」＝忘記身分（見決策 3），與桌面語意不同——已在 UI 文案說明。
  - 密碼強度沒有要求（與桌面 ADR-0067 一致）；Argon2id 參數與原生層相同。
