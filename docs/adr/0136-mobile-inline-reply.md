# 0136. 行動端對話串／回覆：內嵌引用（同 replyTo 資料、平台各自呈現）

- 狀態：已接受（已實作）
- 日期：2026-07-15
- 相關文件：**ADR-0051（對話串：NIP-10 reply e-tag，攜於加密 rumor 內層）**、0133（行動端提及，
  同批行動端對齊）、0095（送出狀態）

## 背景與問題

對話串（ADR-0051）在**桌面**是 Slack 式的**討論串面板**：回覆從主頻道抽出、收攏到根訊息下的獨立
面板（`thread-util.ts` 的 `mainMessages` 把 `replyTo !== undefined` 濾掉）。

**行動端完全沒有**回覆：`ChatMessage.replyTo` 欄位在、後端 `sendMessage`/`sendGroupMessage` 都收
`replyTo`（core `thread.ts` 的 `replyTag`/`threadRoot` 也備好），但行動端 UI **既不能發起回覆、也不
顯示回覆脈絡**——被回覆了也看不出來。

## 決策

### 1. 行動端用**內嵌引用**（Signal/LINE 風），不搬桌面的討論串面板

同一個 `replyTo` 資料，**兩端各自呈現**：桌面＝抽到討論串面板；行動端＝訊息留在原位、上方帶一小塊
**引用預覽**（誰＋摘要）。手機沒有面板的空間與文化，內嵌引用才是原生體驗。這與 ADR-0134 的
`chatBgCss`／`chatBgStyle`（同資料、平台各自取用）同一個取捨。

### 2. 發起回覆：長按選單 →「回覆」→ 引用列 → 送出帶 replyTo

- 任何**未收回**訊息長按都能「回覆」（設 `replyTarget`）。→ 草稿列上方顯示**回覆引用列**（正在回誰＋
  摘要＋✕ 取消）。
- 送出時把 `replyTarget` 當 `replyTo` 交給 `onSend(text, mentions?, replyTo?)`；`MobileApp.send` 轉給
  後端 `sendMessage(..., replyTo)`／`sendGroupMessage(..., replyTo)`（簽名本就支援）。送完清除引用。
- 因此「未收回訊息一律可長按」（過去只有可回應/收回/分享才可長按）——回覆讓長按對每則訊息都有意義。

### 3. 顯示回覆脈絡：泡泡內引用

`m.replyTo` 的訊息，泡泡內上緣顯示被回覆訊息的**引用**（`nameFor` 解出誰＋摘要）。摘要規則與安全：
**已收回的被引用訊息只顯示「已收回」不洩原文**；檔案顯示檔名；長文截斷。引用不到（根訊息已捲出熱區）
就**略過不顯示**，不當掉。回覆訊息本身被收回時整塊引用一併消失（不留殘影）。

## 理由

- 回覆＝帶 NIP-10 reply e-tag 的普通訊息（隨 Gift Wrap 加密，中繼看不到串結構）——這條由 ADR-0051
  保證，本 ADR 只把「發起回覆／顯示引用」補到行動端，未動協定、未動後端。
- 不把桌面的討論串面板硬搬到手機：同資料、平台各自呈現，才不會兩邊都不像原生。純函式
  （`replyTag`/`threadRoot`）已在 core，行動端的內嵌呈現不需要桌面的 `thread-util`（面板聚合），
  故零下沉、零重複。

## 後果

- 正面：
  - 行動端能**發起回覆、看到回覆引用**——與桌面共用同一 `replyTo` 資料、跨端互通（桌面看到的是
    討論串、行動端看到的是內嵌引用，底層同一串）。
  - 引用預覽對已收回/檔案/長文/引用不到都有安全且不崩的處理。
  - 測試 +5（引用顯示、引用不到不顯示、引用已收回只顯示「已收回」、引用檔案顯示檔名、回覆本身
    收回不留引用）。因「未收回一律可長按」更新了 1 個既有圖片分享測試（role=button 不再等於可分享）。
    全綠 mobile 115→120。

- 已知限制：
  - 行動端**不顯示「N 則回覆」的聚合**、也沒有討論串面板——回覆以內嵌引用呈現、留在主時間軸。這是
    刻意的平台差異（桌面才有面板）。
  - 發起回覆的**互動流程**（長按→回覆→引用列→送出）是薄接線，SSR 測試涵蓋引用**呈現**；長按選單本身
    需 jsdom（行動端目前純 SSR，同 ADR-0133 取捨）。
  - 引用只在**根訊息仍在熱區**時顯示；捲出熱區（歸檔）的根訊息引用略過——不特地為引用去讀封存。
